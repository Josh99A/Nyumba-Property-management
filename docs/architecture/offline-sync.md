# Offline synchronization contract

This contract is shared by repositories, the Sembast schema, the sync runner, and Cloud Functions. Connectivity is a hint only: a positive network signal does not mean Firebase or a payment provider is reachable.

## Local records

Every remotely synchronized record stores the domain payload plus sync metadata. Metadata names may be prefixed in persisted JSON to keep them separate from domain fields.

| Field | Meaning |
| --- | --- |
| `id` | Stable aggregate/document ID |
| `serverVersion` | Last applied server aggregate version, or `0` for a local create |
| `syncState` | `synced`, `pending`, `conflicted`, or `rejected` |
| `lastServerUpdatedAt` | Last server timestamp seen; never generated by the client |
| `pendingCommandIds` | Ordered commands currently affecting the record |
| `isDeleted` / `deletedAt` | Server tombstone state |
| `lastError` | Stable error code and safe user-facing context; no secrets |

The local database also contains:

- `outbox`: durable command envelopes and retry state;
- `sync_cursors`: one `(serverUpdatedAt, documentId)` cursor per remote query/scope;
- `session_scope`: current UID, roles, access generation, and active projection scopes;
- optional attachment staging metadata; binary files are stored outside Sembast.

The additive `user_profiles` local store holds editable display/contact fields
and per-user appearance/notification preferences. A profile save writes the
record and a `profile.update`-compatible outbox intent atomically. The active
session may render the optimistic display name immediately, but the UI must
describe the change as pending until a server acknowledgement is available.
Theme selection is a low-risk personal display preference and may take effect
from the local record while that acknowledgement is pending.

Schema migrations are explicit and forward-only. A failed migration must not silently drop outbox records.

The additive local listing schema includes structured public-safe location,
unit facts, amenities/accessibility, availability, lease/cost disclosures,
policies, media references, and optional server projection metadata. Readers
provide safe defaults for records created before these fields existed; no
outbox command is rewritten or discarded. A new publication still validates
the completed structured fields before enqueueing `listing.publish`.

## Local write transaction

For every mutation that may work offline:

1. The application use case validates fields that do not require server truth.
2. It generates one stable `commandId` and any new aggregate IDs.
3. In a single Sembast transaction it applies the optimistic local change, appends the command to `outbox`, records `expectedVersion`, and marks the record `pending`.
4. Repository streams emit the local state immediately.
5. The runner wakes when authenticated and likely online. UI code never performs its own retry loop.

If step 3 fails, neither the local change nor command is retained. Financial, subscription, approval, active-lease, and publication commands may create a local *intent*, but must not change the displayed canonical status to successful/active/published.

## Outbox record

```json
{
  "commandId": "stable UUID/ULID",
  "type": "maintenance.create",
  "schemaVersion": 1,
  "aggregateType": "maintenanceRequest",
  "aggregateId": "generated-offline-id",
  "expectedVersion": 0,
  "payload": {},
  "actorUidAtCreation": "local-session-uid",
  "deviceId": "installation-id-not-ad-id",
  "createdAtLocal": "diagnostic-only ISO timestamp",
  "sequence": 42,
  "attemptCount": 0,
  "nextAttemptAt": null,
  "state": "queued"
}
```

The server ignores `actorUidAtCreation` for authorization and derives identity from `request.auth`. The client refuses to send an outbox item after an account switch; it keeps it quarantined until the original account returns or the user explicitly discards it.

## Push algorithm

- Run at most one sync leader per signed-in account. On web, coordinate tabs when possible; server idempotency still protects against duplicate leaders.
- Preserve FIFO ordering for commands on the same aggregate. Different aggregates may use bounded parallelism.
- Before flushing edits based on remote state, pull deltas. New independent creates can be sent immediately.
- Reuse `commandId` on every retry. Never generate a replacement because of a timeout.
- Treat a timeout or dropped response as `unknown`, then read `commandReceipts/{commandId}` or retry the same command.
- Retry only transient failures (`unavailable`, deadline, rate limit, provider unavailable) with exponential backoff and jitter.
- Do not retry authentication, authorization, validation, entitlement, not-found, or version-conflict failures until state/user input changes.
- A result is complete only when the canonical document/projection has been merged locally. A callable response may include that canonical snapshot; otherwise the subsequent pull supplies it.

Outbox terminal states are `applied`, `rejected`, and `discarded`. Applied records may be compacted after a support/debug window. Rejected records retain the user's input and a stable remediation message.

## Pull algorithm

Foreground screens may use Firestore listeners, while background/reconnect synchronization uses paged queries. Both feed the same merger.

1. Query an authorized scope ordered by `updatedAt`, then document ID, starting after the stored pair.
2. Apply the entire page and advance its cursor in one Sembast transaction.
3. Merge server tombstones instead of deleting immediately.
4. Repeat until fewer than the page limit are returned.
5. Start/refresh listeners from the resulting state for active screens.

Only server timestamps advance cursors. The document-ID tie-breaker prevents records sharing a timestamp from being skipped. A full scope reconciliation is scheduled periodically to detect missed documents, projection removals, and cursor corruption.

When access is revoked, a listener can fail rather than deliver a final removal. The server therefore maintains an access-generation marker in the user's portal root. A changed generation triggers a full reconciliation and removal of local records outside the returned scope. A signed-out offline device cannot learn about a remote revocation; this residual risk is why sign-out/account switch must purge local private data and why local storage encryption matters.

## Merge and conflict policy

Every mutable canonical aggregate has an integer `version`. Commands that edit existing state include `expectedVersion`.

- If versions match, the server applies the mutation and increments the version.
- If they differ, the server returns `VERSION_CONFLICT` plus a safe current snapshot/version.
- The client stores both local intent and server state, marks `conflicted`, and presents a feature-specific resolution.

The listener merge is version-aware and transactional in Sembast. A remote
record whose version is not newer than the locally known synced revision is
ignored. If an aggregate has any durable outbox item, the listener never
overwrites it; a different remote version changes only the local sync metadata
to `conflicted`. The original local record and outbox intent remain available
for an explicit feature-level rebase or discard flow.

Automatic last-write-wins is allowed only for explicitly designated low-risk fields such as a personal display preference or unpublished draft description. It is forbidden for amounts, invoice/payment state, leases, occupancy, approval, subscriptions, entitlements, maintenance status, and listing publication. Append-only items (comments, events, attachments) use their own stable IDs and are naturally mergeable.

If a command is rejected, do not blindly restore an old local snapshot because later commands may depend on it. Rebase later commands on the returned canonical version when the feature policy allows; otherwise mark the chain for user resolution.

## Attachments

Uploads are two-phase:

1. Create an offline attachment intent with local URI, checksum, content type, size, and command ID.
2. When online, upload to `uploads/{uid}/{commandId}/...` under restrictive Storage Rules.
3. Submit/finalize the owning command with the staging object path and checksum.
4. A function validates the object, moves or copies it into a canonical private/public path, writes document metadata, and schedules staging cleanup.

The application must tolerate local files disappearing before upload and surface a recoverable `LOCAL_ATTACHMENT_MISSING` state. Public listing images are never public merely because they were uploaded; only the server-owned public projection/path is readable publicly.

The checked-in demo baseline has no configured Storage uploader. Its listing
picker therefore keeps bounded image data references only inside an unpublished
local draft so the selection survives a local repository reload. Publication
validation fails closed while any such local reference remains. This is a
temporary non-production bridge: the Firebase adapter must replace it with the
attachment-intent flow above (including checksum and missing-file recovery)
before photo publication is enabled. Local image data must never be copied into
`publicListings` or treated as an acknowledged upload.

On Android and iOS, each account-scoped Sembast database uses AES-256-GCM with
a distinct random key stored through Keychain/Keystore-backed secure storage.
Web IndexedDB remains unencrypted because browser JavaScript cannot provide an
equivalent device-keystore boundary; public/browser workspaces must therefore
minimize retained PII. Desktop builds are development-only and currently share
that unencrypted limitation. Account switching closes the previous database
before opening the next account-specific path, so records and outbox commands
are quarantined rather than read across identities.

## Operation policy

| Operation | Offline behavior |
| --- | --- |
| Edit property/unit details | Optimistic local edit; queued command; version conflict possible |
| Create a unit | Local pending unit; server confirms entitlement and atomically reserves unit count |
| Submit maintenance request | Optimistic pending request and attachment intents |
| Draft notice/listing | Optimistic local draft |
| Publish listing | Queue intent, but remain `publicationPending` until server validates availability and entitlement |
| Generate invoice | Queue request; amount/due state appears only from server canonical result |
| Initiate electronic payment | May save an intent offline, but checkout/provider interaction requires online state |
| Confirm payment / issue receipt | Never optimistic; only signed provider webhook or authorized server workflow confirms |
| Approve/suspend landlord | Online admin command; server audit required |
| Change subscription | Online billing workflow; provider webhook is authoritative |

## Failure and observability requirements

- Display `offline`, `syncing`, `pending changes`, `action required`, and `last synced` states without claiming connectivity guarantees.
- Metrics: oldest outbox age, commands by result code, retry count, conflicts, projection lag, pull duration, and full-reconciliation failures.
- Logs include command ID, type, actor UID hash, aggregate ID, result code, and latency. Do not log command payloads, tokens, payment details, exact addresses, or document contents.
- Tests cover process death between local commit/push, duplicate delivery, lost callable response, clock skew, token expiry, permission revocation, account switch, out-of-order pulls, same-timestamp pages, and provider callback duplication.
